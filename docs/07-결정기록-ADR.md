# 결정 기록 (ADR — Architecture Decision Records)

> 작성일: 2026-07-03
> 목적: 주요 설계 결정과 **근거**를 남긴다. 브레인스토밍에서 오간 트레이드오프를 보존해, 나중에·면접에서 그대로 설명할 수 있게 한다. ([06-포트폴리오-전략](06-포트폴리오-전략.md)의 "ADR" 항목)

형식: 상태 / 맥락 / 결정 / 근거 / 영향.

## ADR-001 — 도메인 = 1:1 로봇 축구
- **상태**: 확정
- **맥락**: 실시간 통신 + 로봇 제어를 어필할 최소 게임이 필요.
- **결정**: 11:11 풀 축구 대신 **1:1** (로봇 2 + 물리 공 + 골대 2).
- **근거**: 11:11은 복잡도 과다로 핵심(실시간 통신)이 묻힘. 1:1은 최소이면서 어필 포인트를 다 담음.
- **영향**: 군집 개념 축소 → 개별/계층 제어가 전면.

## ADR-002 — 서버 권위 시뮬레이션, Rust + rapier2d
- **상태**: 확정
- **결정**: 물리·판정은 **서버에서만**(server-authoritative). 서버 = Rust(rapier2d + tokio/axum), 클라 = 브라우저 canvas 2D.
- **근거**: 로봇 무선 제어 서사와 직결. Rust는 시스템 역량 어필 + rapier2d 품질. 사용자 C++/Go 배경으로 감당 가능.
- **영향**: 클라는 렌더만. 치트 방지·디버프 판정이 서버로 집중.

## ADR-003 — 통신: WebSocket + JSON, 30Hz 스냅샷, 보간만
- **상태**: 확정
- **결정**: WebSocket 단일 세션 · JSON · 서버 60Hz 물리 / 30Hz 브로드캐스트 · 클라 **보간(interpolation)만**. 예측/보정 없음.
- **근거**: 1:1 규모엔 보간으로 충분. "가볍게" 목표. 예측/보정은 과함.
- **영향**: 무선 시뮬레이터로 견고성 시연([02](02-네트워크-프로토콜.md) §7).

## ADR-004 — 계층형 제어
- **상태**: 확정
- **결정**: 전략 층 / 행동 층(FSM) / 액추에이터 층. 사람이 개입하는 층으로 모드(직접/전략/AI)를 구분, 런타임 전환.
- **근거**: 실제 로봇 제어 구조(플래너→컨트롤러→액추에이터)와 일치. "다양한 제어" 어필.

## ADR-005 — 파츠 커스터마이징 채택, 커머스 제외
- **상태**: 확정
- **맥락**: 로봇 다양성 심화 + "장착 부위" 아이디어. 메일/유료샵/가챠도 검토됨.
- **결정**: **파츠 조립체**(7슬롯, 결정적 스탯 기여 + 코스메틱, 해금형 로드아웃) 채택. **메일·유료샵·결제·랜덤강화(가챠)는 제외.**
- **근거**: 파츠는 로봇 테마를 살리고 코어를 심화하며 백엔드 없이 가능. 커머스는 계정·DB·영속성을 요구해 초점을 흐리고 완성 리스크↑ → 필요 시 별도 포트폴리오.
- **영향**: 로봇 = 부품 조립. 파츠 카탈로그는 개발자 배포(서버 카탈로그).

## ADR-006 — 식별/영속성 = 매직코드 최소 인증 + SQLite, **Phase 2로 분리**
- **상태**: 확정(방향), 구현은 **Phase 2**
- **맥락**: 파츠 소유·착용을 영속화하려면 사용자 식별과 저장이 필요. 익명 토큰 / 이메일 검증없음 / 매직링크 / 매직코드 / 계정 등을 검토.
- **결정**:
  - 영속성 = **SQLite**(임베디드, 파일 하나). 용도: parts 카탈로그·가격, user_parts(소유), user_loadout(착용), (선택) 재화.
  - 식별 = **패스워드리스 매직코드**(이메일로 6자리 코드 → 사용자가 입력). 링크가 아니라 **코드**라 폰/크로스디바이스에서 호스트 도달 문제 없음.
  - 인증 요구 범위 = **슬롯 참가·파츠 소유/착용에만**. **관전은 익명.**
  - **v1(코어)에는 넣지 않는다.** 코어 완성 후 **Phase 2**로 얹는다.
- **근거**: 매직코드는 비번 해싱·OAuth 없이 실제 소유 검증을 주는 최소 인증. SQLite는 인프라 부담 없이 영속성 제공. 코어(실시간 제어)가 주인공이므로 인증을 뒤로 미뤄 완성도를 지킴.
- **영향 / 필수 준수**:
  - 데이터 정책 변경: v1 인메모리 → Phase 2 SQLite 영속 + 익명→식별.
  - 보안: 코드 **시도횟수 제한 + 짧은 만료(~10분) + 1회용 + 발송 레이트리밋** 필수.
  - 발송: 개발=콘솔 출력(무료), 배포=이메일 API. base URL은 설정값(`PUBLIC_URL`)으로.
  - 재화·구매(경제)는 더 뒤로 미룸 — 우선 "소유·착용 영속화"만.

## ADR-007 — 결정성 범위 = same-build
- **상태**: 확정
- **결정**: 결정적 시뮬은 **같은 빌드·같은 머신** 범위까지만. 크로스 플랫폼 float 동일성은 추구하지 않음.
- **근거**: 크로스 플랫폼 결정성은 난이도 폭증. 리플레이/테스트엔 same-build로 충분. ([06](06-포트폴리오-전략.md) §4)
- **구현 가이드 (how)**:
  - **고정스텝 누산기**: 라이브 루프는 벽시계 `interval`이 아니라 **누적 시간 → 고정 dt로 N스텝** 소비 방식(`accumulator += elapsed; while accumulator >= DT { step(); accumulator -= DT }`). 프레임 지터와 무관하게 스텝수 결정.
  - **sim 순수·I/O 분리**: `step`/`tick`은 입력→상태 순수 함수(소켓·시간·난수 미포함).
  - **난수**: 시드 고정 RNG만(예: 아이템 스폰). 시드는 경기 시작 시 기록 → 리플레이 재현.
  - **순서 결정성**: 로봇/콜라이더 순회를 `HashMap` 이터레이션에 의존하지 않음(정렬된 Vec/고정 인덱스).
  - **rapier**: `enhanced-determinism` 피처 + 고정 timestep, 병렬 솔버 비결정성 주의.
  - **골든 리플레이**: 기록된 입력·시드로 재생 → 스냅샷 해시 비교로 회귀 감지([05 §6](05-개발프로세스.md)). (Plan 2에서 도입)
- **적용 단계**: 누산기·골든 리플레이는 물리 도입 시점인 **Plan 2**부터. Plan 1의 `tick` 함수는 이미 결정적.

## ADR-008 — 핵심 밸런스 원칙: 승리=골, 전투=양념
- **상태**: 확정
- **결정**: **승리 조건은 골뿐**(KO 승리 없음). 전투는 목적이 아니라 **템포·공간을 버는 수단**. **제압형**(쓰러뜨리고 득점)과 **도망형**(공 몰고 회피하며 득점)이 **공존**하며 서로 하드카운터하지 않는다.
- **근거**: 실시간 통신·제어가 주인공. 전투가 축구를 잡아먹으면 정체성이 흐려짐. 3분 클락이 득점 지향을 강제.
- **영향**: 이후 모든 밸런싱의 기준. 스턴 짧게·제거 없음·태클은 공을 떨굼·회피 도구는 유효하게. 전투 피해→정지회복→템포손실이라는 **자체 밸런싱** 루프.

## ADR-009 — 전투 해결 모델
- **상태**: 확정 (수치는 튜닝)
- **결정**:
  - 로봇↔로봇 충돌만 전투 판정, 공은 데미지 없음(순수 물리).
  - 상호 데미지: `impact × (상대 attack합 / 내 defense합) × 피격부위 취약도`, 양쪽 부딪힌 부위에.
  - 효과 = 넉백/스턴/데미지, **선택은 결정적(또는 시드 고정)**, **임팩트 비례 중첩**, **세기 = effect프로필 × impact × 저항 (곱)**.
  - 부위 HP: 0 초과=점진저하+정지회복, **0=파손 다운(행동불능)→일정 시간 뒤 전체 풀 리페어**로 복귀(제거 아님).
  - 공 = **자유 물리 + 밀기 드리블**, 피격 시 공 이탈.
- **근거**: 파츠가 "숫자"를 넘어 **전투 스타일**을 정함(로드아웃=전투 정체성). 결정적 모델로 리플레이/실력 보존([ADR-007]). 파손 다운은 "득점 창"으로 전투를 축구에 환급([ADR-008]).
- **영향**: 파츠에 `attack/defense/hp/effect` 속성 추가([00 §6]). 스냅샷 `event`에 피격부위·효과 정보(이펙트용). 데미지·회복·효과선택은 **순수 함수 → TDD 최적**.
- **구현 addendum (Plan 3b/3c 반영)**:
  - **`impact` = 상대 linvel(post-step) 근사** 채택 — ADR의 "접촉 임펄스"를 간소화. 진짜 임펄스는 `ContactForceEvent`(`CONTACT_FORCE_EVENTS`+threshold) 경로로 추후 상향 가능.
  - **부위별 취약도 항은 미구현**(3b/3c는 로봇 총 attack/defense) — 후속 튜닝.
  - **데미지 = `damage_on_contact`(공/방) + 가산 `effect.damage`(dmg_w)**. 효과 가중치는 카탈로그에 초기값만(밸런싱 부채).
  - 충돌 이벤트는 `(rA,rB,pA,pB)` 정렬 후 적용(멀티히트 결정성). 벽/공 오독은 **로봇 부위 멤버십 셋**으로 차단.

## ADR-010 — 필드 지오메트리(골 입구 펜스 + 코너 챔퍼)
- **상태**: 확정 (KB-43/44, 플레이테스트 반영)
- **맥락**: AI-대-AI/사람 플레이 중 (1) 로봇이 골 입구 틈으로 필드 밖 이탈, (2) 사각 코너에 공이 끼어 빠지지 않음.
- **결정**:
  - 골 입구에 **GOALFENCE 충돌그룹** 펜스 — rapier `InteractionGroups`로 로봇(ROBOT 그룹)은 막고 공(BALL 그룹)은 통과(득점 유지). 그룹: `BALL`/`ROBOT`/`SOLID`/`GOALFENCE`.
  - 4개 코너에 **45° 챔퍼**(빗변 cuboid, SOLID 그룹)로 공 끼임 완화.
- **근거**: 서버 권위 물리([ADR-002])에서 필드 경계는 물리로 강제. 충돌그룹 필터가 "로봇만 가두고 공은 통과"를 선언적으로 표현.
- **영향**: `server/src/physics.rs`의 `groups` 모듈 + `new_kickoff_with`(벽/펜스/챔퍼 생성). 골 판정은 공만 통과하므로 불변.

## ADR-011 — 스태미나/스프린트 최소 슬라이스
- **상태**: 확정 (KB-45). 수치는 튜닝.
- **맥락**: "체력(배터리)" 요청 — 고속 이동은 소모, 저속은 상시. [00-개요-및-게임설계](00-개요-및-게임설계.md) §8 "스태미나/오버히트 모델"은 걷기·달리기·쉬기·오버히트까지 아우르는 목표 모델을 서술하나, 이번 슬라이스는 그중 **걷기/달리기만** 구현한다.
- **결정**:
  - **걷기(`max_speed`)는 상시** 가능. **스프린트(`sprint_speed`, Shift 홀드=`input.run`)는 `stamina>0`에서만**, `apply_controls`에서 게이팅하고 `stamina==0`이면 클라가 `run:true`를 계속 보내도 걷기 속도로 자동 폴백.
  - 스프린트 중 소모 / 아닐 때 재생. `StaminaState`는 **순수 `f32`**(결정성).
  - **제외(YAGNI)**: 오버히트 감속 페널티·AI 스프린트·복잡한 재생 구분.
- **구현 addendum (KB-53)**: 회복 정책을 명확화 — 스프린트=소모, **걷기(이동 입력 thrust≠0)=유지(회복 안 함)**, **가만히 있을 때(이동 입력 없음)만 회복**. (기존엔 스프린트만 아니면 걸으며도 회복 → "쉬면 회복" 감이 없었음. 회전만 하는 것은 이동으로 치지 않음.)
- **근거**: 결정적·순수 유지([ADR-007])로 리플레이 보존. 최소 슬라이스로 "실시간 제어에 자원 관리 한 축"을 시연, 나머지는 부채로 미룸.
- **영향**: `parts.rs`의 `StatSet` +`sprint_speed`/`stamina_max`/`stamina_regen`; 신규 `stamina.rs`; `physics.rs`(게이팅/스냅샷); `world.rs`의 `ControlOutput.run`·`RobotState.stamina`; `session.rs`(run 파싱); [02-네트워크-프로토콜](02-네트워크-프로토콜.md)은 이미 `input.run`+스냅샷 `stamina:0~1` 서술이 있어 정합(무변경). 클라 스태미나바(`render.ts`).

## ADR-012 — 차기(슛) 모델
- **상태**: 확정 (KB-48). 수치는 튜닝.
- **맥락**: 지금까지는 로봇 몸으로 공을 미는 "드리블"만 있어 골 결정력이 로봇 질량/속도에 종속됐다("미니카 축구"). 이미 정의만 되어 있던 `StatSet.kick_power`를 실제로 배선해 축구 정체성(원거리 슛/짧은 패스)을 살린다. 홀드 차징(`shootCharge`, [02](02-네트워크-프로토콜.md) 예약)·공중볼/높이·커브·태클/점프는 이번 슬라이스 범위 밖(YAGNI).
- **결정**:
  - **모드리스 탭 발사**: 별도 모드 전환 없이 Space를 탭하면 그 자리에서 즉시 킥. 클라는 `input.kick`(bool)만 올리고, 서버가 로봇별 이전 값 대비 **false→true 상승엣지에서만 1회** 발사(홀드해도 반복 발사 없음).
  - **세기 = 그 순간의 상하 이동입력 3단**: thrust>0(↑)=1.0(강)·thrust==0=0.75(중)·thrust<0(↓)=0.5(약). 최종 임펄스 크기 = 레벨 × 로봇 `kick_power`(로드아웃 집계, chassis/프리셋 무관하게 파츠 스탯이 결정).
  - **방향 = 그 순간의 좌우 이동입력**: 로봇 정면(rotation) 기준 turn>0(←)=+35°, turn<0(→)=−35°, turn==0=정면(0.61rad 근사).
  - **유효 조건**(하나라도 불충족이면 무동작, 상태 변화 없음): 로봇이 파손 다운·스턴 아님(이동 입력과 동일 게이팅), 쿨다운(`shoot_lock`) 없음, 공이 로봇 정면 부채꼴 사거리(거리 ≤ 약 0.9m, 정면 콘 75° 이내) 안.
  - **쿨다운**: 발사 시 `kick_cd ≈ 0.45초` 설정, 매 스텝 DT만큼 감소(0 클램프). 잔여 중엔 스냅샷 `st`에 `shoot_lock` 태그.
  - **결정적 순수 임펄스**: 넉백과 동일한 `apply_impulse(dir * magnitude, true)` 패턴. 계산 전 과정에 I/O·RNG·HashMap 순회 없음.
- **근거**:
  - 밀기만으로는 축구가 아니라 "질량으로 미는 미니카"에 가까워 정체성이 흐림 — 차기가 있어야 슛/패스라는 축구 고유의 동작이 생긴다.
  - 이동 키(↑↓←→)를 그대로 재사용해 세기·방향을 결정하는 **모드리스 설계**는 키 최소화·직관성을 확보한다(별도 조준/차징 UI 불필요, [01-UX](01-UX-화면구성.md) 원칙과 정합).
  - 순수 임펄스 + 결정적 엣지 판정은 리플레이/재현성을 보존한다([ADR-007]).
- **영향**:
  - `parts.rs`의 `StatSet.kick_power`가 처음으로 실배선됨 — 카탈로그 프리셋에 실값 부여(`striker`=`body-light.kick_power`, `guard`=`body-std.kick_power`; striker가 민첩형이라 더 강함).
  - 프로토콜: `input.kick`(bool, 구현) 추가, `st.shoot_lock`(이미 문서화되어 있던 태그) 실사용 시작.
  - `world.rs`(`ControlOutput.kick`), `session.rs`(파싱), `physics.rs`(`prev_kick`/`kick_cd`/`apply_kicks`/스냅샷), `client/src/input.ts`(Space 키).
  - **YAGNI로 제외**: 홀드 차징(`shootCharge`)·공중볼/높이·커브·태클/점프 — 후속 슬라이스 후보로만 남김.
- **구현 addendum (KB-52)**: AI(`ChaseBallAi`)도 슛한다 — 공이 정면 사거리(0.85m) 안 + 정면이 상대 골 방향으로 정렬됐을 때만(자책골 방지). 초기 "AI 킥 없음"(YAGNI) 결정을 갱신(차기가 생긴 이상 AI도 쏴야 대칭). 상대 골 방향은 팀으로 결정(블루=+x/레드=−x, `check_goal` 규약).

## ADR-014 — UI/비주얼 디자인 방향(미드나잇 프리시전 콘솔 + Neon Telemetry Arena)
- **상태**: 확정 (KB-51). 정적 시안 승인, 실제 클라 적용 진행.
- **맥락**: 웹 화면이 "초기 개발 모드"(흰 페이지 + 검은 캔버스 + OS 기본 버튼 + 디버그 netsim 노출)라 포트폴리오 첫인상에서 크게 손해. 레퍼런스: [refero.design](https://styles.refero.design/)(상위 제품 디자인 시스템 라이브러리), [behance.net](https://www.behance.net/)(게임 UI/HUD).
- **결정**: 화면을 **두 층**으로 분리해 각각 방향을 정함.
  - **웹 크롬(게임 밖) = "미드나잇 프리시전 콘솔"** — Linear/Raycast 계열(refero의 *midnight terminal / precision instrument* 군). netsim/RTT를 디버그 구석에서 **일급 LINK MONITOR 텔레메트리로 격상**, 스코어/타이머/팀명을 캔버스에서 크롬 HUD로 이전.
  - **게임화면(캔버스) = "Neon Telemetry Arena"** — Behance(Rocket League HUD의 절제 + Brawl Stars의 가독성 + FUI 발자국 텔레메트리). **콘솔 팔레트 완전 공유**(신규색 없음), 1m 그리드·좌우 진영 그라데이션·중앙선/서클, 팀색 **글로우 아웃라인 로봇(6족 유지)** + 발자국 링, 팀색 골 글로우 프레임, 공 트레일, HP/스태미나 6px 캡슐, **이벤트색은 앰버 단일**로 절제.
  - **토큰**: 배경 `#0B0E14`/패널 `#151A24`/보더 `#242c3a`, 텍스트 `#E6E9EF`/`#8390a3`, 팀 블루 `#3AA0FF`·레드 `#FF5A6A`, 앰버 `#FFB020`·그린 `#3DDC97`(상태색, 팀색과 분리). 타입: UI=시스템 sans, **데이터/라벨=등폭(계기 readout, tabular-nums)**. **다크 단일 테마(의도적 확정).**
- **근거**:
  - **독립 리뷰어 2패널(각 3렌즈, 서로 안 봄)** 로 결정. 웹 크롬: 시스템/dev-tool·채용 렌즈가 **독립적으로 같은 결론**(계측 콘솔), 방송/아케이드 렌즈의 juice는 "엔지니어 관객에겐 얕게 읽힘"으로 반증돼 그 **명료함(스코어보드)만 흡수**. 게임화면: 아레나(가독성)·FUI(홀로그램)·미니멀(축약) 중 **아레나를 중심**으로, 양 극단의 자기-리스크(성능·가독성 / 6족 정체성·전투 박력 소실)를 회피하고 좋은 요소만 이식.
  - 관객은 엔지니어/채용자 → 계측기 룩이 **실시간·서버 권위·동시성 역량을 30초에 신호**([ADR-002]·[ADR-013]과 시각 언어 일치). netsim/RTT 격상은 넷코드 작업(Plan 7a)을 보이는 자산으로 전환.
- **영향**: `client/index.html`(콘솔 레이아웃+CSS), `client/src/render.ts`(캔버스 아트), `client/src/main.ts`·`devpanel.ts`(크롬 HUD 배선). **기능 불변**(보간·ping/RTT·netsim·참가·키입력). 정적 시안(Artifact): `claude.ai/code/artifact/6accd3ca-38d2-4896-8528-b28b826ba7b8`.
- **제외/후속**: 강한 방송 juice·파티클 과다·라이트 테마는 배제. 게임 흐름 상태(대기→선택→결과) 화면은 Plan 5.

## ADR-013 — 넷코드 견고성 시연 슬라이스(보간 + 링크 시뮬 + RTT)
- **상태**: 확정 (KB-50, Plan 7a). 수치는 튜닝.
- **맥락**: 문서는 "무선 로봇 제어 시뮬레이터"를 표방했지만([02](02-네트워크-프로토콜.md) §7·[ADR-003]) 실제론 지연/지터/드랍·보간·RTT가 없어 그 서사가 **화면에서 증명되지 않았다.** 포트폴리오의 차별점(실시간 통신·서버 권위)을 "보이게" 만드는 최소 슬라이스.
- **결정**:
  - **클라 보간(interpolation)**: 스냅샷을 버퍼링하고 렌더 시각을 최신보다 ~0.1s 뒤로 둔 뒤 두 스냅샷 사이를 보간(pos 선형, rot **최단호**). 스냅샷 `time`(서버 sim 초)을 타임스탬프로 사용. 예측/보정 없음([ADR-003] 준수), 보간 on/off 토글.
  - **서버 아웃바운드 링크 시뮬**: 연결별로 `state`·`pong` 송신에 지연(`delay_ms`)+지터(`0..jitter_ms`)+확률 드랍(`drop_pct`)을 주입. 개발 패널(`netsim` 업링크)로 실시간 조절. 전송 큐는 `deliver_at` 순 배출(head-of-line 블로킹 없음).
  - **ping/RTT**: `ping`↔`pong`(같은 id 에코)로 왕복 측정, 화면 표시. 퐁도 링크심을 통과해 **RTT가 설정 지연을 반영**.
  - **결정성**: 링크심은 전송 계층 wrapper(자체 xorshift PRNG). 물리 `step`/스냅샷 생성 경로엔 관여하지 않음 → 리플레이/골든 해시 불변([ADR-007]).
  - **제외(YAGNI)**: 인바운드(업링크) 링크심·재연결/슬롯 유예·클라 예측/보정·대역폭 최적화 — 후속(Plan 7 잔여).
- **근거**: 보간이 없으면 어떤 견고성 작업도 "덜덜거림"으로 묻히고, 링크심 토글이 없으면 넷코드 품질이 리뷰어 눈에 안 보인다. 이 셋을 묶어야 "지터 켜면 덜덜 → 보간 켜면 매끄럽고, 지연+드랍에도 플레이 가능"을 실제로 시연할 수 있다. 순수 전송 계층 분리로 결정적 코어를 지킨다.
- **영향**: `server/src/net.rs`(연결별 링크심·`drain_due`·`parse_net_ctrl`), `client/src/net.ts`·`interp.ts`·`devpanel.ts`·`main.ts`·`index.html`. 프로토콜: `ping`/`pong`/`netsim` 추가([02](02-네트워크-프로토콜.md) §3). 게임 sim·`session::Uplink`는 불변(넷 제어는 net 계층 전용 파싱).

## ADR-015 — 팀 로스터: 팀당 2대(공격형+방어형) + 협동 AI
- **상태**: 확정 (KB-57). 배치·수치는 튜닝.
- **맥락**: 1v1(2대)은 12×8 필드가 너무 넓고 밀기 위주가 됐다. 팀당 **공격형(striker)+방어형(guard)** 2대(총 4대)로 공간을 채우고 협동 플레이를 만든다. "모델 선택"은 **로스터 고정**으로 대체 — 경기 중 프리셋 교체(물리 바디 재생성)의 복잡도를 피한다.
- **결정**:
  - **고정 로스터 4대**: index 0=Blue striker(공격·사람 조종 가능), 1=Blue guard(수비·AI), 2=Red striker, 3=Red guard. `RobotState.id`는 팀값(2대 공유). 프리셋은 기존 `striker`/`guard` aggregate(속도차 내장: striker 최고속↑).
  - **협동 = 역할 분담**(GameView/프로토콜 확장 없이): `ChaseBallAi`(공격)=공 추적+정렬 시 슛; `DefenderAi`(수비)=자기 골–공 직선 위, 자기 골에서 `DEFENDER_GUARD_DIST(2.5m)` 이내 지점을 지킴+클리어 슛. 목표가 근본적으로 달라 **공에 뭉치지 않는다.** 스턱탈출·슛(자책골 회피) 판정은 공용 자유함수로 공유.
  - **친선 데미지 금지**: 같은 팀 충돌은 데미지/넉백/스턴 모두 차단(`teams[ra]!=teams[rb]`일 때만 적용).
  - **사람은 팀 striker만 조종**(guard 항상 AI 동료). leave/AFK 시 striker는 공격 AI로 복귀.
  - 2대 레거시 생성자(`new_kickoff_with`/`new_kickoff`)는 `#[cfg(test)]`로 보존(수많은 물리 테스트 불변). 실행은 신규 `new_match()`(4대). `reset_kickoff`는 저장된 `kickoff_layout`을 써 로봇 수 무관.
- **근거**: 역할 분담만으로 협동을 얻어 GameView 확장·모델 선택의 복잡도를 회피(YAGNI). 로스터 고정이 외형(KB-56)·속도 비대칭을 자연히 살린다. 결정적 코어·리플레이 불변.
- **영향**: `world.rs`(`MATCH_KICKOFF`), `physics.rs`(`teams`/`kickoff_layout`/`build`/`new_match`/친선필터), `control.rs`(`DefenderAi`+공용 함수), `main.rs`(4슬롯·팀→striker·일반화). 클라: `interp.ts`·render 보행 상태를 **인덱스 기반 매칭**(팀 id 비유일 대응), 배지는 striker 슬롯(Blue=0/Red=2). 스냅샷 robots/ctrl 길이 4, `check_goal`·팀 스코어 불변.
- **후속**: 사람의 조종 로봇 전환·2대 모두 사람·모델 커스터마이징은 후속 슬라이스.
