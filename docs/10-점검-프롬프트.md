## 1. 착수 전 — 설계·계획 드라이런 점검

이대로 지으면 막힘/공백이 없는지 확인(코드 생기기 전).

```
docs/와 docs/superpowers/plans/<계획파일>.md를 기준으로,
이대로 구현했을 때 나올 결과물을 전체 점검해줘. 파일 근거를 들어 표로:
1) 최종 산출물 형태와 실행법(서버/클라, 명령)
2) 테스트가 무엇을 어떻게 검증하나 + 커버리지 공백
3) 개발환경 재현성(의존성/버전/빌드/실행이 문서만으로 되나)
4) 설계→구현 사이 빠진 연결(타입·인터페이스 불일치, 미정의 참조, 문서 간 모순)
5) 착수 전 보완할 위험요소
결론: "이대로 착수 가능한가"에 대한 판정.
```
→ 독립 검증이 필요하면 `Plan` 서브에이전트로 실행.

## 2. 구현 중 — 중간 상태 점검 (매 카드/에픽 후)

어디까지 왔고 건강한가.

```
KANBAN.md와 Plan 문서(docs/superpowers/plans/...)를 기준으로 현재 상태를 점검해줘:
1) 완료(Done)/진행(In Progress)/남은(Backlog) 카드 — 계획 대비 어디까지 왔나
2) 빌드·테스트 건강도: cargo test / vitest 통과 여부, 실패·경고
3) 방금 구현이 해당 카드의 테스트를 실제로 통과시키는지(증거)
4) 설계 문서(docs/)와 실제 코드의 드리프트 — 어긋난 곳
5) 바로 다음 착수할 카드와 그 선행조건
표로 요약하고, "지금 건강한가"에 대한 판정.
```

## 3. 구현 중 — 진행 방향 확인 (에픽 경계·의사결정 지점)

계획이 여전히 맞나, 틀면 어디로.

```
지금까지 구현을 근거로 진행 방향을 재점검해줘:
1) 남은 Plan(2~8) 순서가 여전히 최적인가 — 재정렬/분할/병합할 것
2) 구현하며 드러난 사실이 설계(docs/·ADR)를 바꿔야 하나 — 있으면 ADR 갱신 제안
3) 스코프 드리프트 신호(계획에 없던 것 추가/누락)
4) 다음 에픽 착수 전 보완할 위험요소
5) "지금 방향 유지 vs 조정" 권고 1개
근거 파일을 들어서. 결론은 명확한 권고로.
```

## 3.5 오케스트레이션 모드 (싼 모델 분업)

큰 작업은 **오케스트레이터(메인 세션)가 상황관리**하고, 실무는 **역할별 서브에이전트**로 분업(subagent-driven). 비용 절감을 위해 역할별 모델을 명시:

| 역할 | 담당 | 모델 |
|---|---|---|
| 상황관리(오케스트레이터) | 메인 세션 | 세션 모델(Opus) — 조율·판단, 컨텍스트 보존 |
| 신규개발·수정 | implementer 서브에이전트 | **Sonnet** (잘 명세된 TDD에 충분) |
| 완료 결과 확인 | verifier 서브에이전트 | **Sonnet/Haiku** (테스트 재실행·스펙 대조) |
| 전체 확인·최종 정리 | final integrator | **Opus** (마일스톤 종합 판단) |

원칙: **일상 작업은 싼 모델, 최종 판단만 유능한 모델.** 싼 모델이 놓친 건 verify/최종리뷰 층이 잡음 → 비용↓, 품질 유지. 까다로운 작업(외부 라이브러리 API 등)만 모델 상향.

## 4. 구현 후 — 산출물 검증 (완료 주장 전)

실제로 도는가.

```
Plan N을 구현한 diff와 실행 결과를 점검해줘:
- cargo test / vitest 통과 여부(출력 증거)
- cargo run + npm run dev로 실제 경기/기능이 보이는지
- 05 §7 DoD 항목 대비 충족/미충족
- 남은 리스크
```
관련 슬래시 커맨드:
- `/verify` — 앱을 띄워 엔드투엔드 동작 관찰
- `/code-review` — 작업 diff의 버그·단순화 리뷰
- `/security-review` — 보안 관점(네트워크/입력 검증)

## 6. 참고 — 자동으로 걸리는 체크포인트

프롬프트를 매번 안 쳐도 실행 스킬에 점검이 내장됨:
- **subagent-driven-development**(권장 실행): 태스크마다 서브에이전트 실행 후 2단계 리뷰
- **executing-plans**: 배치 실행 + 체크포인트
- **verification-before-completion**: "완료" 주장 전 실제 검증 강제
- **requesting-code-review / /code-review**: diff 리뷰
