## 1. 목적 (왜 만드나)

**포트폴리오용** 프로젝트. 어필하려는 핵심 역량:

1. **웹 클라이언트 ↔ 서버 실시간 통신** (가장 중요)
2. 그 실시간 통신 구조가 **로봇 무선 제어(텔레메트리/명령 루프)와 동일**하다는 스토리
3. **계층형 로봇 제어**(전략 → 행동 → 액추에이터)
4. **서버 권위(server-authoritative) 시뮬레이션**
5. 게임 기본기: 캐릭터 선택, 시작/종료, 승패, 보상(랭킹)

분야 결합: **게임 + 로봇**. 목표 태도는 "가볍게 이것저것 해보기" — 최소 구성으로 위 어필 포인트를 모두 담는다.

## 2. 한 줄 컨셉

> 서버가 1:1 로봇 축구 경기를 권위 있게 물리 시뮬레이션하고, 브라우저 클라이언트가 무선 링크(WebSocket)로 접속해 로봇을 **직접/전략/AI** 제어한다. 아무도 조작하지 않을 땐 AI끼리 자동 경기(어트랙트)로 계속 돌아간다.

### 프레이밍: "엔진 + 게임 하나"

이 프로젝트는 단순한 미니게임이 아니라 **재사용 가능한 실시간 권위 게임 서버 프레임워크를 만들고, 그 위에 게임 하나(로봇 축구)를 얹은 것**으로 본다. 프레임워크 축:

- **결정적 시뮬레이션 코어** (I/O 분리, 고정 timestep, 리플레이/테스트 가능)
- **데이터 주도 카탈로그** (파츠·로봇·효과·아이템·**게임 종류**를 설정으로 정의 → 코드 변경 없이 확장)
- **멀티 게임 프로토콜** (`gameType`을 실어 특정 게임에 묶이지 않음)
- **Controller 추상화** (인간·AI·전략을 하나의 인터페이스로 통일 — §9)

→ 이 렌즈에서 보면 확장성(멀티게임·카탈로그·페이즈)은 스코프 크립이 아니라 **의도된 플랫폼 설계**다. 로봇 축구는 그 프레임워크의 **첫 번째 게임(vertical slice)**.

## 4. 확정 기능 목록

- 1:1 로봇 축구: 로봇 2대 + 물리 공 1개 + 골대 2개, 사각형 박스 비주얼 (골 입구 펜스·코너 챔퍼로 이탈/끼임 방지, [ADR-010](07-결정기록-ADR.md#adr-010--필드-지오메트리골-입구-펜스--코너-챔퍼))
- 3분 경기, 골 수로 승/무/패
- 성능이 서로 다른 **로봇 8종** (초기 2종 오픈, 6종 잠금 — 추후 추가)
- **어트랙트 모델**: 경기가 항상 라이브로 진행, 슬롯을 이어받아 참가
- **계층형 제어**: 전략 / 행동(FSM) / 액추에이터, 런타임 전환, 사람 미개입 층은 AI
- 액션: 방위·이동·스톱·달리기·쉬기·슛(+세기)·태클·점프(회피 대시)
- 자원: 스태미나 / 오버히트 → 감속·쉬기
- 디버프(전부 서버 판정): 이동불가·슛불가·정지불가·태클불가·점프불가·감속 + 후딜·쿨다운
- 아이템: 서버가 주기적으로 1개 배치, 접촉 획득, 초기 2종(부스트/방해), 확장 가능
- **로봇별 승률 랭킹**: 인메모리 집계, 서버 재시작 시 리셋, 별도 DB 없음
- 스택: Rust(rapier2d + WebSocket) 서버 / 브라우저 canvas 2D 클라이언트

## 5. 게임 규칙

- **경기 시간**: 3분(180초). 종료 시 골 수로 판정.
- **결과**: 승 / 무 / 패.
- **랭킹**: **로봇 캐릭터별 승률**을 집계해 높은 순으로 표시.
  - 집계 단위는 로봇 캐릭터 (v1은 계정 개념 없음).
  - 저장(v1): 서버 메모리만. **서버 재시작 = 리셋**. DB 없음.
  - **Phase 2**: 파츠 소유·착용 영속화를 위해 SQLite + 매직코드 식별 도입([07 ADR-006](07-결정기록-ADR.md)). 코어 완성 후 별도 페이즈.
- **AI 경기(어트랙트) 집계 여부**: 승률 왜곡을 막기 위해 **사람이 최소 1명 참여한 경기만 랭킹에 반영**하는 것을 기본으로 한다. *(확정 필요)*

## 6. 로봇 (캐릭터)

- 총 **8종**. 초기 **2종 오픈**, 나머지 **6종 잠금**("Coming soon") — 추후 스탯 세트만 추가하면 노출.
- 로봇마다 **성능(스탯)이 다르다**. 선택 화면의 설명글이 곧 이 스탯 성향.

### 스탯 축 (물리 파라미터로 표현)

| 스탯 | 의미 | 영향 액션 |
|---|---|---|
| `maxSpeed` | 최고 이동 속도 | 이동 |
| `accel` | 가속력 | 이동 |
| `turnRate` | 회전 각속도 | 방위 |
| `mass` | 질량(몸싸움/넉백 저항) | 충돌·태클 |
| `kickPower` | 슛 임펄스 최대치 | 슛 |
| `sprintSpeed` | 달리기 속도 | 달리기 |
| `stamina` | 스태미나 용량 | 달리기·오버히트 |
| `cooling` | 스태미나/발열 회복 속도 | 쉬기 |
| `tackleForce` | 태클 임펄스 | 태클 |
| `recovery` | 후딜/스턴 회복 속도 | 디버프 |
| `jump` | 회피 대시 거리/쿨다운 | 점프 |

### 로봇 = 파츠 조립체 (모듈러 로드아웃)

로봇은 **부위(파츠)의 조립체**이며, **총 스탯 = 부위별 스탯 기여의 합**이다. 로봇이 부품으로 조립된다는 주제와 맞고, 위 "성능 다른 로봇"을 부위 조합으로 심화한다.

**장착 슬롯 (7개)**

| 슬롯 | 개수 | 주 기여 스탯(예) |
|---|---|---|
| 머리 | 1 | `recovery`(감지/판단) |
| 목 | 1 | `turnRate` |
| 몸통 | 1 | `mass`, `stamina`, `cooling` |
| 앞다리 | 2 | `accel`, `tackleForce` |
| 뒷다리 | 2 | `maxSpeed`, `sprintSpeed` |
| 꼬리 | 1 | `jump`(균형/회피) |

- **파츠 정의** = 슬롯 + 이동 스탯 기여(델타) + **전투 속성** + **부위 HP** + 외형(코스메틱). 새 파츠 추가 = 정의 하나 추가.
- **8종 로봇** = 미리 짜둔 **프리셋 로드아웃**(2 오픈 / 6 잠금). 플레이어는 해금된 파츠로 조합을 **커스터마이즈**할 수 있다.
- **코스메틱 전용 파츠**(스탯 0, 색/모양만)도 허용 — 사각형 프리미티브라 렌더 가볍다.
- **범위 제한(확정)**: **계정·결제·DB·랜덤 강화(가챠) 없음.** 파츠는 결정적 스탯 기여 + 해금형 로드아웃일 뿐. 근거·경계는 [06-포트폴리오-전략](06-포트폴리오-전략.md) §3 참조.

**파츠의 전투 속성** (§12 전투 모델에서 사용)

| 속성 | 의미 |
|---|---|
| `attack` | 공격 기여(합산). 충돌 시 상대에게 주는 효과 크기 |
| `defense` | 방어 기여(합산). 피격 시 받는 효과 경감 |
| `hp` | 부위 내구도. 0이면 파손 다운([§12](#12-전투--데미지-모델)) |
| `effect` | 효과 프로필: 넉백/스턴/데미지 성향 가중치 (예: 둔중=넉백↑, 강타=스턴↑, 날카로움=데미지↑) |

→ 로봇의 **총 공격/방어 = 파츠 속성 합**, 효과 성향 = 파츠 `effect` 조합. 즉 **로드아웃이 전투 스타일을 결정**한다.

## 7. 액션셋 (액추에이터 층)

| 액션 | 물리 동작(rapier2d) | 키(직접모드, 잠정) | 관련 스탯 |
|---|---|---|---|
| 방위 | 회전 (각속도/토크) | ← → | `turnRate` |
| 이동 | 전진 추력 (↓ 후진) | ↑ / ↓ | `accel`, `maxSpeed` |
| 스톱 | 적극 제동 → 속도 0 | S | 제동력 |
| 달리기 | 일시 가속 + 발열/스태미나 소모 | Shift(홀드) | `sprintSpeed`, `stamina` |
| 쉬기 | 제자리 회복(빠름) | R(홀드) | `cooling` |
| 슛 + 세기 | 홀드로 충전 → 릴리즈 임펄스 | Space(홀드=세기) | `kickPower` |
| 태클 | 순간 돌진 임펄스(공/상대 밀침) | X | `tackleForce` |
| 점프 | **회피 대시**(짧게 도약, 태클 회피) | Z | `jump` |

> 점프 = **회피 대시(A안)** 로 확정. 탑다운 2D(평면)라 진짜 높이(z축)는 없으며, 태클↔스턴↔점프가 서로 물리는 상성으로 게임성을 만든다. *(대안 B: 가짜 높이 홉 / C: 점프 제외 — 확정 필요 시 재검토)*

### 슛 세기(charge)
Space를 누르는 동안 충전 게이지가 차오르고, 떼는 순간의 세기로 킥 임펄스가 결정된다. 로봇별 `kickPower`가 상한.

## 8. 스태미나 / 오버히트 모델

```
[걷기]  기본 속도, 발열 없음, 스태미나 서서히 회복
  │ Shift 홀드
[달리기]  빠름, 스태미나 소모 / 발열 상승
  │ 스태미나 0 / 과열
[오버히트]  달리기 잠김 + 이동 감속(디버프)
  │ 쉬기 / 정지
[쉬기]  제자리 빠른 회복 → 스태미나 참 → 복귀
```

- **스톱**: 관성 미끄러짐을 즉시 멈추는 적극 제동(조작 스킬).
- **쉬기**: 멈춰서 스태미나/발열을 빠르게 회복. 오버히트 후 사실상 필수.
- 전략/AI 모드에서는 FSM이 과열 전에 쉬기를 자동 판단.

## 9. 계층형 제어 (Control Hierarchy)

```
        오퍼레이터 / AI 입력
   ┌─────────┼───────────────┐
 [전략 층]  "공격/수비/공추적/대기" 또는 목표점 지정 (무엇을 할지)
   ▼         ▼
 [행동 층]  FSM: "공까지 이동 → 슛각 정렬 → 킥" (목표 → 동작 변환)
   ▼         ▼
 [액추에이터] thrust / turn / kick impulse / tackle / jump (물리 명령)
   ▼         ▼
        rapier2d 물리 시뮬 (서버)
```

**핵심**: 같은 스택에서 **사람이 어느 층에 개입하느냐**로 제어 모드가 갈린다. 사람이 안 잡은 층은 AI/자동 로직이 채운다.

| 모드 | 개입 층 | 아래 층 | 입력 |
|---|---|---|---|
| **직접** | 액추에이터 | (우회) | 키보드(탱크 드라이브) |
| **전략** | 전략 | 행동·액추에이터 = AI 자동 | 마우스/버튼(터치) |
| **AI** | 없음 | 전 층 자동 | 없음(관전) |

- **런타임 전환** 가능(경기 중 직접↔전략↔AI 토글) — "다양한 제어" 어필의 핵심.
- 행동 층은 **간단한 FSM**으로 구현: `Idle`, `ChaseBall → AlignShot → Kick`, `Defend → InterceptBall`. (behavior tree는 범위 밖)

### Controller 추상화 (아키텍처 주춧돌)

이 계층을 코드로 묶는 것이 **`Controller` 인터페이스**다 — 인간 입력·전략 지시·AI가 **모두 같은 인터페이스**를 구현한다.

```rust
trait Controller {                    // 인간/AI/전략/스크립트 공용
    fn decide(&mut self, view: &GameView) -> ControlOutput;
}
```

- 제어 모드 = 어느 층에 어떤 Controller를 꽂느냐. 사람이 안 잡은 층은 AI Controller가 채움 → 런타임 전환이 인터페이스 교체로 자연스럽게 성립.
- 이득: **SOLID 경계** · 확장성(새 AI/제어 = 구현 추가) · **테스트**(고정 입력 **스크립트 Controller**로 게임 로직을 결정적으로 검증).
- 상세: [09-AI-설계](09-AI-설계.md). 이 추상화는 프레임워크의 핵심 축(§2 프레이밍).

## 10. 디버프 / 상태 (전부 서버 판정)

디버프는 **"액션 잠금 + 스탯 변경"** 으로 일반화한다.

| 상태 | 발생 | 효과 |
|---|---|---|
| 강제 대기(후딜) | 태클/슛/점프 직후 | 짧게 다음 액션 잠금 |
| 이동불가(스턴) | 태클 당함 / 큰 충돌 | 조작 불가 + 넉백 |
| 슛불가 | 특정 디버프/아이템 | 슛 잠금 |
| 정지불가 | 특정 디버프/아이템 | 제동 불가(계속 미끄러짐) |
| 태클불가 / 점프불가 | 쿨다운/디버프 | 해당 액션 잠금 |
| 감속 | 오버히트/디버프 | 이동 속도 저하 |
| 쿨다운 | 태클/점프 사용 후 | 재사용 대기 |

- 서버 스냅샷의 로봇 필드에 상태 목록(`st`)과 쿨다운(`cd`)을 담아 브로드캐스트.
- **서버 권위 시연 포인트**: 스턴/잠금 중 클라 입력이 와도 **서버가 무시**. 골·아이템·승패 전부 서버 판정.

## 11. 아이템 시스템 (최소 구성)

```
경기 중 서버가 일정 주기(기본 20초)마다 필드에 아이템 1개 배치(한 번에 하나)
→ 로봇이 물리적으로 접촉하면 획득(서버 감지) → 효과 적용
```

- 획득은 **물리 접촉으로 서버가 판정** → 클라→서버 별도 명령 불필요.
- 스냅샷에 아이템 위치/종류 포함, 클라는 렌더만.
- **초기 2종** (확장 가능):
  - 🟢 **부스트/회복**: 획득 시 자기 스태미나 즉시 회복 + 잠깐 스피드업
  - 🔴 **방해**: 획득 시 상대에게 잠깐 랜덤 디버프(슛불가/감속 등)

## 12. 전투 / 데미지 모델

> **핵심 밸런스 원칙**: 승리 조건은 **골뿐**. 전투는 그 자체가 목적이 아니라 **템포·공간을 버는 양념**이다. 두 스타일이 공존한다 — **제압형**(부딪혀 쓰러뜨리고 득점) / **도망형**(공만 몰고 회피하며 득점). ([07 ADR-008](07-결정기록-ADR.md))

### 충돌 = 상호 데미지 (부위별)
- **로봇↔로봇** 충돌만 전투 판정. **로봇↔공 / 공↔벽**은 순수 물리(차기·밀기), 데미지 없음.
- 부위 식별 = **복합 콜라이더**(단일 강체 + 파츠별 콜라이더에 태그).
- 산출: `impact(접촉 임펄스) × (상대 attack합 / 내 defense합) × 피격부위 취약도` → **양쪽 모두** 부딪힌 부위에 적용(상호).

### 효과 = 넉백 / 스턴 / 데미지 (확정)
- **선택은 결정적**(또는 시드 고정) — 리플레이·실력 기반. ([07 ADR-009](07-결정기록-ADR.md))
- **임팩트 클수록 중첩**: 약한 접촉=데미지 소량 / 강한 태클=넉백+짧은 스턴+데미지+**공 탈취**.
- **세기 = 세 요소 곱**: `파츠 effect 프로필 × impact × 피격부위 저항`.
- 스턴은 **짧게**(창 열기용), 감금 아님. 넉백은 위치 이동. 데미지는 부위 HP 차감.

### 부위 HP · 파손 다운 · 회복
- **0 초과(손상)**: 부위 스탯 기여가 점진 저하. **정지(스톱/쉬기) 시에만 회복**, 이동 중 회복 없음.
- **HP 0(파손)**: **파손 다운** — 로봇 행동불능·정지(조작 X, 물리 몸체로는 존재해 밀리고 공이 튕김) → **일정 시간 뒤 부품 전체 풀 리페어**(같은 로드아웃, HP 전량 복구)로 복귀. **제거·리스폰 아님.**
- "파손 다운"(비자발) ≠ "스톱"(자발 액션) — 구분.
- 다운 시간 = 상대의 **득점 창** = 전투가 축구로 환급되는 지점. (지속시간은 밸런싱 튜닝)

### 공 컨트롤 (확정)
- **자유 물리 공 + 밀기 드리블**: 공을 "붙잡지" 않고 몸으로 밀어 몰기. 태클/넉백/스턴 당하면 공이 떨어져 나감. → "공 잡고 도망"=공을 앞에 두고 이동.

### 자체 밸런싱
- 데미지→회복하려면 멈춰야 함→공을 못 쫓음→**템포 손실**. 즉 전투 피해가 **간접적으로 득점 기회 손실**로 환산되어, 별도 페널티 없이 "전투=양념" 톤이 유지된다.
