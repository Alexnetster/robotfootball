## 1. 실행 형태

가장 단순한 시연 구성:

```
[노트북 1대]
  └─ Rust 서버 프로세스 1개 (물리 시뮬 + WebSocket)
        ▲                          ▲
        │ WebSocket                │ WebSocket
   [브라우저 창 A]             [브라우저 창 B]
   (BLUE 오퍼레이터)           (RED 오퍼레이터)
```

- **상시 서버 불필요**: 시연할 때만 `cargo run` + 브라우저 열기.
- 물리 객체가 소수(로봇 2, 공 1, 벽, 골, 아이템)라 **CPU/메모리 부담이 극히 작음** — 일반 노트북에서 여유.
- 창 2개 대신 한쪽/양쪽을 AI로 두면 무인 데모(ATTRACT).

---

## 2. 클라이언트 요구사항

- **모던 브라우저**(Chrome/Edge/Firefox/Safari 최신): WebSocket + Canvas 2D + requestAnimationFrame 지원이면 충분.
- 플러그인/설치 없음. URL만 열면 됨.
- 데스크톱: 키보드(직접) + 마우스(전략).
- 모바일(확장): 터치(전략 모드) 또는 관전. 반응형 레이아웃([01](01-UX-화면구성.md) §6).

## 3. 네트워크 접속 경로

| 경로 | 방법 | 용도 |
|---|---|---|
| **로컬** | `localhost:PORT` | 개발·단일 머신 시연 |
| **LAN** | 서버 머신 IP `ws://192.168.x.x:PORT` | **폰/다른 기기에서 접속**(같은 와이파이) |
| **외부(선택)** | 터널(ngrok/cloudflared) 또는 배포 | 원격 시연 |

- **폰 접속**: 같은 와이파이에서 서버 머신 IP로 접속 → 브라우저에서 바로 관전/전략 플레이. 실제 무선 지연·지터가 자연스럽게 발생.
- **크로스 디바이스 시연**: 데스크톱(직접) vs 폰(전략) 1:1 → 무선 링크 + 다양한 제어를 한 번에 어필.
- 외부 노출·정식 배포는 범위 밖(§6 참고).

## 4. 데이터 / 상태 정책

### v1 (코어)
- **인메모리만**: 경기 상태·승률 랭킹 모두 서버 메모리에 보관.
- **서버 재시작 = 리셋**: 랭킹 초기화. 영속 저장 없음.
- **DB 없음**, 외부 API 없음 → 운영·보안 부담 최소.
- 계정/로그인 없음(관전·플레이 모두 익명). 집계 단위는 로봇 캐릭터.

### Phase 2+ (선택 확장 — 코어 완성 후)
- **SQLite 영속** + **패스워드리스 매직코드 식별**(파츠 소유·착용). 관전은 계속 익명.
- v1 코어에는 넣지 않는다. 상세(스키마·보안·발송·`PUBLIC_URL`)는 **[08-future-backlog](08-future-backlog.md)**, 근거는 [07 ADR-006](07-결정기록-ADR.md).

## 5. 시연 시나리오 (권장)

1. 서버 실행 → 브라우저 열기 → **ATTRACT**: AI vs AI 라이브 데모("DEMO") 관전.
2. 키 입력 → **로봇 선택**(스탯 다른 로봇 프리뷰) → Enter로 참가(AI 로봇 이어받기).
3. 경기 중 **모드 스위처**로 직접↔전략↔AI 전환 시연.
4. 두 번째 창(또는 폰)으로 반대 슬롯 참가 → **1:1 / 크로스 디바이스** 대결.
5. **NET SIM** 켜서 지연 200ms/드랍 주입 → 보간·재연결로 견디는 모습 시연.
6. 3분 종료 → 승/무/패 → **로봇별 승률 랭킹** 갱신 확인.

## 6. 배포 (선택 · 범위 밖)

정식 공개가 필요해질 때의 참고안(현재 구현 대상 아님):

- **클라(정적)**: Cloudflare Pages / Vercel 등에 빌드 산출물 배포.
- **서버(상시)**: 무료/저가 VPS(예: Oracle Cloud Always Free) 1대에 Rust 서버 상주, 앞단 Cloudflare.
- HTTPS/WSS, 도메인 연결은 이때 검토.

> 포트폴리오 시연 목적에는 **로컬/LAN 실행으로 충분**하며, 배포는 필요 시 확장한다.

## 7. 비기능 요구 (NFR)

목표 수치(1:1 규모 기준, 구현 중 계측·조정):

| 항목 | 목표 |
|---|---|
| 물리 tick | 60Hz 고정, tick 처리 **p99 < 3ms** (여유 큼: 바디 소수) |
| 스냅샷 브로드캐스트 | 30Hz, 세션당 payload 대략 < 2KB(JSON) |
| 종단 체감 지연 | LAN/로컬 < 50ms + 보간 지연버퍼 ~100ms |
| 클라 렌더 | 60fps 목표, 드랍 프레임 최소 |
| 동시 세션 | v1 목표 = 경기 2 슬롯 + 관전 소수(수~수십). 대규모 아님 |
| 메모리 | 서버 프로세스 경량(수십 MB 수준) |
| 서버 재시작 | 진행 중 경기·인메모리 랭킹은 **소실**(설계상 허용). 재시작 시 어트랙트부터 재개 |

**브라우저 매트릭스**: Chrome/Edge/Firefox/Safari 최신 2개 버전. 요구 기능 = WebSocket · Canvas 2D · requestAnimationFrame. 폴리필 없음.

**성능 검증**: 헤드리스 AI vs AI 벤치([09 §6](09-AI-설계.md))로 tick 시간·payload를 계측, 관측성 오버레이([06](06-포트폴리오-전략.md))로 실측 표시.
