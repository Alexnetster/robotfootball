# 네트워크 / 프로토콜

> 작성일: 2026-07-03
> 관련: [00-개요-및-게임설계](00-개요-및-게임설계.md), [01-UX-화면구성](01-UX-화면구성.md)
> 프레이밍: **클라 ↔ 서버 = 로봇 무선 통신 링크** (다운링크=텔레메트리, 업링크=명령)

---

## 1. 링크 모델 (무선화 관점)

```
   [웹 클라이언트]                          [Rust 서버 = 로봇 본체 / 관제]
        │  ── 업링크: 명령(제어 의도) ─────────►  │
        │  ◄──── 다운링크: 텔레메트리(상태) ──────  │
        └─────────── WebSocket 단일 세션 ──────────┘
```

- **다운링크(텔레메트리)** = 30Hz 상태 스냅샷 (로봇/공/디버프/아이템/스코어)
- **업링크(명령)** = 제어 입력(액션/전략/모드전환), **변화 시** 전송
- 실제 로봇의 "관제가 상태를 계속 받고, 조종 명령을 쏜다"와 동일 구조.

---

## 2. 전송 방식

- **WebSocket 단일 채널**, 메시지 인코딩 = **JSON**.
- 근거: 1:1 · 30Hz 규모엔 대역폭·성능 문제 없음. JSON은 디버깅/가시성이 좋아 포트폴리오에 유리.
- WebSocket은 TCP 위 → **무손실·순서보장**(진짜 무선의 손실은 없음). 이를 보완하는 것이 §7 무선 시뮬레이터.
- *(심화, 범위 밖)*: 진짜 비신뢰 전송이 필요하면 **WebTransport(UDP 계열)** 로 교체 가능. 지금은 도입하지 않음.

## 3. 메시지 카탈로그

### 3.1 다운링크 (서버 → 클라)

| 타입 | 시점 | 내용 |
|---|---|---|
| `welcome` | 접속 시 | 클라ID, 현재 경기 상태, 설정(tickRate 등), **카탈로그 버전**, **게임 종류(현재+가능 목록)** |
| `catalog` | 접속 시/갱신 | **데이터 주도 정의**: 파츠(슬롯·스탯·`attack/defense/hp/effect`·코스메틱·**price**)·프리셋 로봇·효과·아이템·**게임 종류 정의** (§4.2) |
| `state` | 30Hz | 스냅샷 (아래 §4) |
| `event` | 발생 시 | **경기 시작(종류·총시간)** / **경기 종료(결과)** / 골 / 아이템 / **피격(부위·효과·임팩트)** / 파손 다운 / 리페어 (§4.1) |
| `ranking` | 경기 종료·갱신 | 로봇별 승률 테이블 |
| `error` | 요청 거부 시 | `{ code, msg, ref? }` — join 거부(슬롯 점유·잘못된 로드아웃)·미소유 파츠·잘못된 메시지 등 |
| `pong` | ping 응답 | 지연 측정용 타임스탬프 |

### 3.2 업링크 (클라 → 서버)

| 타입 | 내용 | 빈도 |
|---|---|---|
| `join` | `{ "slot":"blue"|"red", "loadout": <로드아웃> }` 슬롯 이어받기. 로드아웃 = 프리셋 id 또는 슬롯별 파츠 id (§4.3) | 단발 |
| `leave` | `{}` 나가기 → AI 인계 | 단발 |
| `input` | `{ "fwd":bool, "back":bool, "turn":-1|0|1, "run":bool, "kick":bool, "stop":bool, "rest":bool, "shootCharge":bool, "tackle":bool, "jump":bool }` 액션층 | 변화 시 스트리밍 |
| `strategy` | `{ "mode":"attack"|"defend"|"chase"|"hold" }` 또는 `{ "target":{ "x":number, "y":number } }` | 단발/저빈도 |
| `ctrlmode` | `{ "value":"direct"|"strategy"|"ai" }` 런타임 제어 전환 | 단발 |
| `loadout` | `{ "slots": { "head":"h1", "body":"b2", ... }, "cosmetic": {...} }` 커스터마이즈 저장/장착 (§4.3) | 단발 |
| `ping` | `{ "t":"ping", "id":number }` 지연 측정(구현됨, KB-50) — 서버가 같은 `id`로 `pong` 에코 | 1Hz |
| `netsim` | `{ "t":"netsim", "delay_ms":number, "jitter_ms":number, "drop_pct":0..100 }` 개발용 링크 시뮬 설정(구현됨, KB-50) | 변경 시 |

**예약(향후 페이즈, v1 미구현)** — 타입만 예약해 두어 나중에 추가해도 v1과 충돌 없음:
`auth`(Phase 2 매직코드) · `purchase`(Phase 3 샵) · `mail_read`/`mail_claim`(Phase 4 메일).

- `input.kick`(KB-48, **구현됨**): 모드리스 탭 발사 플래그. Space 키다운 시 `true`, 키업 시 `false`.
  서버가 false→true 상승엣지에서만 1회 발사(ADR-012). `shootCharge`(홀드 차징)는 아직 미구현인
  후속 설계 자리표시자로 유지(YAGNI — 현재는 세기를 이동입력 상하로 대체).
- **넷코드 견고성**(KB-50, **구현됨**, [ADR-013](07-결정기록-ADR.md)): 스냅샷 `time`(초)을 클라 **보간**
  타임스탬프로 사용(렌더를 ~0.1s 뒤로 두고 두 스냅샷 사이 선형/최단호 보간). `netsim`으로 서버가
  **아웃바운드(state·pong)** 에 지연/지터/드랍을 주입(무선 링크 시뮬), `ping`/`pong`으로 RTT 측정(퐁도
  링크심을 통과해 RTT에 반영). 인바운드 링크심·재연결·클라 예측은 범위 밖(YAGNI).
- **슬롯 참가 UX**(KB-55, **구현됨**): 스냅샷에 `ctrl`(슬롯별 `"human"`|`"ai"`) 방출(설계상 `state.robots[].ctrl`를 구현은 `GameState.ctrl[]` 배열로). 참가 버튼은 **토글**(내 팀=나가기/타 팀=전환/무참가=참가). 사람 점유 슬롯이 **30초간 Input 없으면 AFK 자동 해제**→AI. 세션 식별자는 미도입 — 클라가 자기 슬롯을 낙관적 추적 후 `ctrl`로 재조정(데모 스코프 한계 허용).

모든 메시지는 `t`(타입) 필드로 구분. 순서/지연 판별용 `seq` 또는 타임스탬프를 부가할 수 있다. **미지의 `t`는 무시**(전방 호환).

## 4. 상태 스냅샷 스키마 (`state`)

```json
{
  "t": "state",
  "gameType": "soccer_1v1",
  "phase": "attract" | "playing" | "ended",
  "time": 142,
  "duration": 180,
  "score": { "blue": 1, "red": 0 },
  "robots": [
    {
      "id": "blue",
      "loadout": { "head":"h1", "neck":"n1", "body":"b2", "foreleg":"fl3", "hindleg":"hl2", "tail":"t1" },
      "x": 3.1, "y": 1.2, "rot": 0.7,
      "vx": 0.4, "vy": 0.0,
      "ctrl": "human" | "ai",
      "down": { "broken": false, "repairIn": 0.0 },
      "stamina": 0.82,
      "parts": { "head":1.0, "neck":1.0, "body":0.6, "foreleg":0.3, "hindleg":1.0, "tail":1.0 },
      "st": ["stun", "slow"],
      "cd": { "tackle": 0.4, "jump": 0.0 }
    },
    { "id": "red", "loadout": {...}, "x": 8.4, "y": 2.0, "rot": 3.1, "ctrl": "ai",
      "down": { "broken": true, "repairIn": 2.3 }, "stamina": 1.0,
      "parts": { "head":1.0, "body":0.0, "...":1.0 }, "st": ["downed"], "cd": {} }
  ],
  "ball": { "x": 5.0, "y": 1.5, "vx": 0.3, "vy": -0.1, "heldBy": null },
  "item": { "kind": "boost" | "jam" | null, "x": 5.0, "y": 2.0 }
}
```

- `parts` = **부위별 HP 비율**(0.0~1.0). 0이면 파손. 스탯 기여는 이 비율로 점진 저하([00 §12](00-개요-및-게임설계.md)).
- `down` = 파손 다운 상태: `broken`(행동불능 여부), `repairIn`(전체 리페어까지 남은 초). 다운 중이면 `st`에 `downed`.
- `st` = 활성 상태/디버프: `stun`·`shoot_lock`·`stop_lock`·`tackle_lock`·`jump_lock`·`slow`·`downed` 등.
- `cd` = 액션별 쿨다운 잔여(초). `stamina` = 0~1 비율.
- `ball.heldBy` = 자유 물리 공이라 항상 `null`(소유 없음). 밀기 드리블 모델이라 필드만 예약(향후 옵션 대비).
- 좌표는 서버 물리 월드 단위(예: 미터). 클라가 canvas 픽셀로 스케일.

### 4.1 이벤트 스키마 (`event`)

단발 이펙트/사운드용. 예:
```json
{ "t":"event", "kind":"match_start", "gameType":"soccer_1v1", "duration":180 }
{ "t":"event", "kind":"match_end",   "gameType":"soccer_1v1", "score":{"blue":2,"red":1}, "result":{"blue":"win","red":"lose"} }
{ "t":"event", "kind":"hit", "by":"blue", "on":"red",
  "part":"body", "effects":["knockback","stun","damage"],
  "impact": 0.87, "dmg": 0.4 }
{ "t":"event", "kind":"goal",   "by":"blue", "score":{"blue":2,"red":1} }
{ "t":"event", "kind":"broken", "who":"red", "part":"body" }
{ "t":"event", "kind":"repair", "who":"red" }
{ "t":"event", "kind":"item",   "who":"blue", "item":"boost" }
```
- `hit` = 어느 부위에 어떤 효과(넉백/스턴/데미지)가 얼마의 임팩트로 들어갔는지 → 클라 이펙트. **판정은 서버**, 이건 표시용.
- **모든 `event`와 `state`는 접속한 모든 세션에 브로드캐스트** — 공격자·피격자·관전자가 동일 정보를 받는다(§5 참조). 즉 "내가 준 데미지·골"이 상대 화면에도 그대로 갱신됨. "내 정보/네 정보"가 갈리지 않는 게 서버 권위의 핵심 이점.

### 4.2 카탈로그 스키마 (`catalog`) — 데이터 주도

파츠·로봇·효과·아이템 정의를 서버가 내려줌 → 클라 하드코딩 없음, **설정만 바꿔 확장**.
```json
{
  "t": "catalog", "version": 7,
  "parts": {
    "b2": { "slot":"body", "stats":{"mass":1.2,"stamina":0.9}, "attack":8, "defense":12,
            "hp":100, "effect":{"knockback":0.6,"stun":0.3,"damage":0.4},
            "cosmetic":{"shape":"box","color":"#39f"}, "price":0, "locked":false }
  },
  "robots": { "striker-01": { "loadout": {"head":"h1","body":"b2","...":"..."}, "locked":false } },
  "items":  { "boost": {"effect":"stamina_refill+speed"}, "jam": {"effect":"random_debuff"} },
  "gameTypes": {
    "soccer_1v1": { "name":"로봇 축구 1:1", "players":2, "durationSec":180, "locked":false }
    /* 향후: "soccer_2v2", "obstacle_race", ... 정의만 추가하면 노출 */
  }
}
```
- `price`·`locked` 필드는 v1에서 표시(잠금/준비중)만 하고, 실제 구매·소유 검증은 Phase 3에서 사용.
- `gameTypes` = **게임 종류 레지스트리**. 새 게임 = 정의 하나 추가(데이터 주도). v1은 `soccer_1v1`만, 나머지는 잠금 표시.

### 4.3 로드아웃 표현 (`join`/`loadout`)
```json
// 프리셋으로 참가
{ "t":"join", "slot":"blue", "loadout": { "preset":"striker-01" } }
// 커스텀 조합으로 참가/변경
{ "t":"loadout", "slots": {"head":"h1","neck":"n1","body":"b2","foreleg":"fl3","hindleg":"hl2","tail":"t1"},
  "cosmetic": {"body":"skin-red"} }
```
- 서버가 로드아웃 → 복합 콜라이더 + 총 스탯/전투속성 재구성. 미소유·잠금 파츠는 서버가 거부(Phase 2부터 소유 검증).

### 4.4 월드 규격 / 상수 (정본)

좌표·단위·상수의 **단일 소스**. 클라·서버·테스트가 이 값을 공유(설정으로 노출).

| 상수 | 값(초안) | 비고 |
|---|---|---|
| 좌표계 | 원점 = 필드 중앙, +x=오른쪽, +y=위, 각도=라디안 | 물리 월드 단위 = 미터(m) |
| 필드 크기 | 예: 12m × 8m | 튜닝 대상 |
| 골대 폭/위치 | 예: 폭 2.4m, 좌우 끝 중앙 | |
| 물리 tick | 60Hz (dt = 1/60) | 고정 timestep |
| 스냅샷 rate | 30Hz (2 tick마다) | |
| 보간 지연버퍼 | ~100ms | 클라 |
| 경기 시간 | 180s | `state.duration` |

- 클라는 m→canvas 픽셀로 스케일링. 수치는 밸런싱 중 조정하되 **한 곳(설정 상수)** 에서만 바꾼다.

> **구현 단계 주석**: [Plan 1(걷는 뼈대)](superpowers/plans/2026-07-03-walking-skeleton.md)은 이 스키마의 **축소판**(robots·ball·score·time만)으로 시작한다 — loadout·parts·down·st·cd·gameType·duration·phase·item은 각 기능 Plan에서 채운다. 또한 **클라 보간은 Plan 2+에서 도입**하며, Plan 1은 최신 스냅샷을 그대로 렌더한다(30Hz 계단현상 허용).

## 5. 처리 방식 (동기화)

```
서버 루프:
  60Hz 물리 tick — 매 tick:
    버퍼된 최신 입력 적용 → rapier2d step → 디버프/충돌/골/아이템 판정
  30Hz 브로드캐스트 — 2 tick마다 전체 스냅샷을 모든 세션에 전송

클라 루프:
  입력 샘플링(프레임마다) → 변화분만 업링크 송신
  스냅샷 2개를 버퍼 → (now - 지연버퍼 ~100ms) 시점으로 보간(interpolation) 렌더
```

- **입력 버퍼링**: 업링크는 비동기 도착 → 서버는 각 슬롯의 "가장 최근 입력"을 다음 tick에 적용.
- **보간만**: 클라는 예측(prediction)/보정(reconciliation) 없이 두 스냅샷 사이를 lerp. 지연/지터가 있어도 부드럽게. (1:1 규모엔 충분, "가볍게" 목표에 부합)
- **서버 권위**: 물리·충돌·디버프·골·아이템·승패 전부 서버 판정. 스턴/잠금 중 업링크가 와도 서버가 무시.
- **브로드캐스트 = 단일 진실**: `state`·`event`는 **모든 세션에 동일 전송**. 공격자가 준 데미지·골이 상대·관전자 화면에도 그대로 갱신됨 → "내 정보/네 정보" 불일치가 원천적으로 없음(P2P 대비 서버 권위의 핵심 이점).
- **공용 vs 개인 데이터**: 경기 세계(state/event)는 브로드캐스트. **개인 데이터**(지갑·소유·메일)는 해당 세션에만 유니캐스트 — Phase 2+에서 도입.

### 5.1 보안 / 입력 검증 (서버는 클라를 신뢰하지 않는다)

- **모든 업링크 검증**: 스키마·범위·상태 적합성(예: 스턴/다운 중 input 무시, 남의 슬롯 조작 거부, 잘못된 로드아웃/미소유 파츠 거부). 위반은 `error`로 응답 또는 조용히 무시.
- **권위는 서버에만**: 위치·데미지·골·소유는 클라가 못 정함. 클라 값은 요청(intent)일 뿐.
- **레이트리밋**: 업링크 폭주(특히 향후 `auth` 발송) 스로틀. 비정상 빈도 세션 차단.
- **악성/기형 메시지**: 파싱 실패·미지 `t`는 안전하게 무시(크래시 금지). 크기 상한.
- Phase 2+ 인증·소유 검증 상세는 [08-future-backlog](08-future-backlog.md).

## 6. 연결 수명주기 (무선 = 끊김 대비)

```
접속 → welcome 수신 → 관전(다운링크만)
  join → 슬롯 제어(업링크 추가)
  [끊김 감지] 1초 heartbeat(ping) 무응답 N회 → 클라에 "재연결 중" 표시
  자동 재연결 시도 → 서버는 해당 슬롯을 5초 유예 보관
       재연결 성공 → 슬롯 복귀
       유예 초과   → AI가 슬롯 인계
  leave / 세션 종료 → AI 인계 → ATTRACT 복귀
```

- HUD의 **● connected / tick / ping** = 링크 품질계기판(01 §2.3).
- 슬롯 유예 보관은 어트랙트 모델과 맞물림: 사람이 빠지면 경기가 끊기지 않고 AI가 이어감.

### 하트비트 / 킵얼라이브 (ping의 두 역할)

`ping`(1Hz)/`pong`은 **지연 측정 + 생존 확인** 두 목적을 겸한다.

- **유휴(idle) 대비**: `input`은 변화 시에만 보내므로, 관전 중이거나 가만히 있으면 클라 업링크가 침묵한다. → **1Hz ping이 "살아있음" 신호**를 채워 연결 유지 + NAT/프록시의 유휴 타임아웃 방지.
- **양방향 생존 확인**:
  - 서버→클라: **30Hz `state` 브로드캐스트 자체가 하트비트**. state가 끊기면 클라가 서버 이상 감지 → "재연결 중".
  - 클라→서버: **ping**. 서버가 N초간 ping 없으면 세션을 죽은 것으로 보고 슬롯 5초 유예 후 AI 인계.
- **(선택) 전송 레벨 WS ping/pong 프레임**(RFC6455)을 병행하면 프록시/방화벽 keepalive에 유리. 앱 레벨 ping(지연·논리 생존)과 상호 보완.

## 7. 무선 현실성 시뮬레이터 (포함)

WebSocket(TCP)은 실제로는 무손실·순서보장이라, "무선 링크가 열악한 조건에서도 견딤"을 보여주기 위한 **개발/시연용 주입기**.

```
┌ NET SIM ────────────────┐
│ 지연   [——●———] 200ms    │
│ 지터   [—●————]  50ms    │
│ 드랍   [●—————]   5%     │
│ [ 켜기 / 끄기 ]           │
└─────────────────────────┘
```

- 메시지를 렌더/처리 전에 인위적으로 **지연 / 지터 / 드랍** 주입(전송 파이프에 삽입).
- 시연 효과:
  - 지연·지터 → **보간 덕에 화면은 부드러움**(보간 가치 증명)
  - 드랍 → 스냅샷이 한둘 빠져도 보간이 메꿈
  - 심하게 → "재연결 중" → 자동 복구(링크 견고성 증명)
- 평소 꺼둠. 시연 시 "지연 200ms 걸어도 이만큼 버팁니다"를 라이브로 시연.
- 구현 위치: 클라측(수신 스냅샷/송신 입력에 딜레이·드랍) 이 가장 간단. 서버측 per-connection 주입도 가능.
- 참고: 폰이 실제 와이파이/셀룰러로 붙으면 **진짜 지연·지터**가 생겨 시뮬레이터 없이도 자연스러운 무선 시연이 됨. 시뮬레이터는 극한 조건을 임의로 거는 용도.

## 8. 요약 (확정 사항)

- 전송: WebSocket 단일 세션, JSON
- 다운링크: `welcome`/`catalog`(데이터 주도 정의) + 30Hz `state` 스냅샷 + 단발 `event`(피격·골·파손·리페어·아이템)/`ranking`/`pong`
- 업링크: `join`(로드아웃)/`leave`/`input`/`strategy`/`ctrlmode`/`loadout`/`ping`
- 스냅샷에 **게임종류(`gameType`)·총시간(`duration`)·부위별 HP·파손 다운 상태** 포함, `event`에 **경기 시작/종료·피격 부위·효과(넉백/스턴/데미지)·임팩트**
- `state`·`event`는 **전 세션 브로드캐스트**(단일 진실) → 상대·관전자도 동일 갱신. 멀티 게임 대비 `gameType` 실음.
- 서버 60Hz 물리 tick, 30Hz 브로드캐스트, 클라 보간(예측/보정 없음)
- 재연결 5초 유예 → AI 인계, heartbeat로 링크 품질 표시
- 무선 시뮬레이터 포함(개발/시연용)

## 9. 확장성 / 전방 호환 (지금 준비, 나중 구성)

프로토콜이 **지금 설계 전부를 데이터로 실을 수 있고**, 향후 페이즈를 붙여도 v1이 안 깨지도록 준비한다.

- **데이터 주도**: 파츠·로봇·효과·아이템·가격·**게임 종류**는 코드가 아니라 **`catalog`(§4.2)** 로 정의. 새 파츠/로봇/코스메틱/게임 = 서버 카탈로그(설정)만 갱신 → 클라 재배포 불필요. `welcome.catalogVersion`으로 클라가 갱신 여부 판단.
- **멀티 게임 대비**: 메시지에 `gameType`을 실어 프로토콜을 특정 게임에 묶지 않음. `state.gameType`·`match_start.gameType`으로 클라가 어떤 게임/규칙인지 판별 → 다른 게임(2:2, 장애물 코스 등) 추가 시 프로토콜 골격 재사용.
- **타입 기반·미지 타입 무시**: 모든 메시지는 `t`로 구분하고, 클라·서버 모두 **모르는 `t`는 무시**. → 새 메시지 추가가 기존과 충돌하지 않음.
- **예약된 향후 타입** (v1 미구현, 자리만 확보 — 상세 스키마는 [08-future-backlog](08-future-backlog.md)):
  - **Phase 2** — 인증: `auth`(매직코드 요청/검증)
  - **Phase 3** — 샵: `purchase` / `wallet` / `inventory`
  - **Phase 4** — 메일: `mail_list` / `mail_read` / `mail_claim` / `mail_delete` / `mail_badge`
- **소유·잠금 필드 선반영**: `catalog`의 `locked`/`price`, `loadout` 검증 훅을 v1부터 스키마에 두되, v1은 "전부 개방/표시만", Phase 2~3에서 **서버 소유 검증**을 켠다.
- **버전 협상**: `welcome`에 프로토콜/카탈로그 버전 → 불일치 시 클라가 최신 `catalog` 재요청.
