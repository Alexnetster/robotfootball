# 로봇 축구 데모 — 포트폴리오 프로젝트 브레인스토밍 노트

## 1. 목적 (왜 만드나)

- **포트폴리오용** 프로젝트.
- 어필하고 싶은 핵심 역량:
  1. **웹 클라이언트 ↔ 서버 실시간 통신** (가장 중요)
  2. 그 실시간 통신 구조가 **로봇 제어에도 그대로 쓰인다**는 메시지
  3. **게임 기본기**: 방 선택, 시작/종료, 보상 획득
- 분야 결합: **게임 + 로봇**.

---

## 2. 수렴된 컨셉 (현재 초안)

> **소규모 로봇 축구 데모** — 서버가 경기를 권위 있게(server-authoritative) 시뮬레이션하고, **웹 클라이언트 2대**(각각 한 팀 담당)가 실시간 접속. 각 클라이언트는 **자동(AI) 또는 수동(오퍼레이터 전략 지시)** 으로 동작.

### 핵심 메커니즘
- **실시간 통신**: 서버가 정해진 tick마다 경기 상태를 두 클라이언트에 브로드캐스트, 클라이언트 명령이 왕복.
  - → "실제 로봇 텔레메트리/명령 루프와 동일 구조"라는 스토리로 연결.
- **오퍼레이터 조작 = 전략 지시** (개별 미세조작 X). 예: 공격/수비/압박 같은 팀 단위 전략.
- **군집 + 개별 행동 둘 다 필요**:
  - 군집 = 팀 로봇들의 협응(포메이션, 역할 분담)
  - 개별 = 각 로봇의 행동(공 쫓기, 패스, 수비)
- **멀티클라이언트 시연 방법**: 사람 2명 매칭 대신 **브라우저 창 2개**를 띄우거나, 한쪽/양쪽을 **자동(AI) 모드**로 실행 → 매치메이킹·로비 부담 제거.

### 게임 기본 루프
방 선택 → (카운트다운) 시작 → 경기 → 종료(스코어) → 보상 획득

### 로봇 비주얼 (UX)
- 우선은 **사각형 프리미티브**로만: **사각형 몸통 + 사각형 다리 + 사각형 목 + 그 위 머리**.
- 포인트는 **"어떻게 움직이는지"를 보여주는 것** (이동/방향전환 애니메이션).

## 4. 미결 사항 (이어서 결정할 것) — ⭐ 여기서부터 재개

1. **팀 규모** (다음에 바로 결정할 질문)
   - 1:1 → 가장 단순하지만 **군집이 사라짐**
   - **3:3 (추천)** → 군집+개별 둘 다 살리며 관리 가능한 규모, 전략 지시가 의미 있어짐
   - 2:2 → 절충안
2. **렌더링 방식**: 2D 탑다운(canvas, 전략·위치가 명확) vs 3D(three.js 박스, 움직임이 더 멋짐)
3. **전략 지시의 구체 단위**: 포메이션/모드(공격·수비·압박)만? 역할 배정·세트플레이 트리거까지?
4. **로봇 개별 행동 로직**: 규칙기반(FSM/behavior tree) vs 간단한 목표지향 로직 (RL은 범위 밖으로 추천)
5. **기술 스택**: 서버(예: Node + WebSocket / Socket.IO), 물리(서버측 간단 2D 물리), 클라이언트(렌더 라이브러리)
6. **네트워킹 심화 여부**: 상태 스냅샷 브로드캐스트만? 아니면 예측(prediction)·보정(reconciliation)·보간(interpolation)까지 보여줄지 (포트폴리오 가산점 요소)

## 5. 검토했던 다른 후보 (기각/보류)

| 후보 | 설명 | 판단 |
|---|---|---|
| 11:11 풀 축구 | 정식 규모 로봇 축구 | ❌ 복잡도 과다, 핵심(실시간 통신) 묻힘 |
| 장애물 코스/타임어택 | 단일 로봇 코스 통과 + 기록 보상 | 🔵 가장 단순. 축구로 확장 가능한 대안으로 보류 |
| 1:1 미니 축구 | 로봇 1:1 + 공, 한 골 승부 | 🔵 현재 컨셉의 최소 버전 |
| 텔레오퍼레이션 샌드박스 | 실시간 조종 + gait 시뮬 | 🔵 "로봇 제어" 스토리 최강, 게임성 약함 |

→ 최종적으로 **소규모(3:3 유력) 로봇 축구 + 2클라이언트/자동 모드**로 수렴 중.
